# 開発ワークフロー

## 基本方針

- ウォーターフォール型 + V字モデルに基づくテスト設計
- 設計先行：設計書完成後に実装開始
- TDD：Red → Blue → Green の順番でテストケース先行で必ず実装すること（t-wadaのTDD / Kent Beck原著に基づく）

## Skill呼び出しルール

設計ドキュメントの**新規作成・大幅修正**時は、必ず対応するskillを呼び出すこと。

| 作業 | 必須skill |
|------|-----------|
| 要件定義書の作成・修正 | `/requirements` |
| 基本設計書の作成・修正 | `/system-design` |
| 詳細設計書の作成・修正 | `/feature-spec` |
| 作業見積もりの追加 | `/feature-spec-estimate-cost` |

## 実装開始条件

- feature-spec全セクション記載済
- 単体テストケース記載済・ユーザー承認済
- バックエンド機能: openapi.yaml更新済

## タスク実行フロー

### 1. チケット確認・作成

- **Jiraチケットがない場合** → 設計書確認 → チケット構成をユーザーに提案 → 承認後に作成

### 2. DAG整理・計画

- **チケット開始時** → development-plan.md + 詳細設計書を確認
- 着手するチケット群の**依存関係（DAG）を整理**してユーザーに提示する
- **ユーザーの承認を得てから**エージェントを起動する
- 依存のないチケット群は **TeamCreate + Task(general-purpose)** で並列実行する

### 3. テストケース合意

- 詳細設計書に**全ての単体テストケースを記載してから**ユーザーに提示・**承認を得る**（仕様の不明点は `/ask` を使用）
- **テストケース記載後、カバレッジマトリックスで自己レビューしてからユーザーに提示する**
- 実装中に新たなテストケースが必要な場合は、**詳細設計書に追加→承認→実装**の順で進める
- **テストと実装を同時にエージェントに渡してはならない。必ず以下の3フェーズに分けて実行すること。**

#### カバレッジマトリックス（提示フォーマット）

テストケース一覧と共に以下の表を必ず作成し、観点の網羅性を示すこと：

| テストケース | 正常系 | 異常系 | 境界値 | 分岐網羅 |
|------------|--------|--------|--------|----------|
| TC-1: 〇〇 | ✓ | | | ✓ |
| TC-2: 〇〇 | | ✓ | | ✓ |

**自己レビューチェックリスト**（提示前に確認する）：
- [ ] 正常系：主要な成功パスがカバーされているか
- [ ] 異常系：エラーハンドリングが網羅されているか
- [ ] 境界値：入力の端値・null・空文字が考慮されているか
- [ ] 分岐網羅：if/switch の全分岐がカバーされているか

### 4. TDD実装サイクル（Red → Blue → Green）

> **このサイクルは「承認済みテストケースのコード化」を行う段階です。新たなテストケースが必要な場合は、詳細設計書に追加して承認を得てから続けてください。**

#### サイクルの定義

| フェーズ | 内容 |
|---------|------|
| **Red** 🔴 | 詳細設計書のテストケースを「ひとつずつ」コード化。コンパイルエラー・アサーション失敗の状態 |
| **Blue** 🔵 | テストを通すための**最小限の実装**を書く。汚くても良い、とにかく通す |
| **Green** 🟢 | リファクタリングでコード品質を改善する。テストは引き続き通る |

#### エージェントへの指示分割（必須）

| フェーズ | エージェントへの指示 | 完了条件 |
|---------|------------------|---------|
| **Red** | テストファイルのみ作成。実装は一切書かない | テストが失敗する（コンパイルエラー or アサーション失敗）・レビュー承認済 |
| **Blue** | テストを通す最小限の実装を書く | 全テスト通過・カバレッジ80%以上 |
| **Green** | リファクタリング（必要な場合のみ） | テストが引き続き通る・コード品質改善済 |

### 5. テストケースレビュー

- 詳細設計書のテストケースと実装されたテストコードを突き合わせ、全ケースが網羅されているかをユーザーに提示・確認する

### 6. 完了前レビュー → コミット

- 下記「レビュー手順」に従い、ユーザー承認後にコミットする

### 7. チケットクローズ

- 「やったこと」記載、ステータス更新

## エージェントチーム運用

**実装タスクは必ずエージェントチームに委譲する。自分（コーディネーター）はコードを書かない。**

### エージェント命名規則

エージェント名は `{役割}_{チケットID}` の形式とする（アンダースコア区切り）。

例: `backend_KAN25`, `frontend_KAN29`, `openapi_KAN28`

### 並列化の判断

| 条件 | 実行方式 |
|------|---------|
| チケット間に依存なし | 並列（エージェントチーム） |
| 先行タスクの成果物が必要 | 順次（先行完了後に起動） |

## 補足ルール

### マイグレーション変更時のseeds更新ルール

**マイグレーションファイルを作成・編集した場合は、必ずseedsも更新すること。**

| 項目 | 規則 |
|------|------|
| 対象 | `backend/migrations/*.sql` の作成・編集 |
| 必須アクション | 関連する `backend/seeds/*.sql` の更新・追加 |
| 適用対象 | テストデータやマスタデータとして使用するテーブル |
| 例外 | ログテーブル、一時テーブルなどseedsが不要なテーブル |

**手順**:
1. マイグレーションファイル作成後、関連するseedsファイルを更新
2. `make seed` で正常に適用されることを確認
3. マイグレーションとseedsを一緒にコミット

**例**:
- `000002_create_users.up.sql` 作成 → `seeds/02_test_users.sql` 作成
- テーブル追加 → テストデータ用seeds追加

### エピックブランチ運用ルール

**親チケット（エピック）開始時にブランチを作成し、子チケットは全て同じブランチで作業する。**

| 項目 | 規則 |
|------|------|
| ブランチ命名 | `ticket/{チケット番号}`（例: `ticket/36`） |
| 作業タイミング | 親チケット（エピック）の作業開始時 |
| 適用範囲 | エピック配下の全子チケット |
| ブランチ削除 | PRマージ後削除（mainブランチ保護） |

**手順**:
1. 親チケット着手時: `git switch -c ticket/{親チケット番号}`
2. 子チケット作業: 全て同じブランチで実装・コミット
3. エピック完了時: `gh pr create` でPR作成、レビュー後にマージ

## レビュー手順

> **作業完了後は必ずレビューを求めること。チケット単位で順番に実施する。**

### レビューの流れ（1チケットごと）

1. 作業完了後、下記フォーマットでレビューを提示する
2. ユーザーの承認を得る
3. 承認後にコミットする

### 提示フォーマット（必須）

**レビューを求める際は必ずこのフォーマットを使用すること。**
単体テストの追加・変更がある場合「カバレッジ」セクションを設ける

```
## [チケットID] チケット名

### 概要
（このチケットで何を・なぜ変更したかを2〜3行で説明）

### 変更ファイル一覧
- path/to/file.go — 変更の目的
- path/to/other.ts — 変更の目的

### 詳細
**path/to/file.go**
- 変更点1
- 変更点2
```
