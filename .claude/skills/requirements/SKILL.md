# Requirements Skill

## 概要

新規バージョンの基本設計として、`docs/versions/<version>/requirements.md` に要件定義と機能一覧を作成するSkillです。

**位置づけ**: 詳細設計（feature-spec）の前段階。このドキュメントで定義された機能が、詳細設計の対象となる。

## 必須入力

このSkillを呼び出す際は、以下の情報を提供してください：

- **バージョン番号**（例: `1_0_0`, `1_1_0`）
- **背景・目的**（なぜこのバージョンが必要か）
- **実現したい機能の概要**（箇条書きでOK）

## 出力ファイル

```
docs/versions/<version>/requirements.md
```

例: `docs/versions/1_0_0/requirements.md`

## 出力テンプレート

```markdown
# バージョン X.X.X 要件定義

## 1. 背景・目的

### 1.1 背景
{このバージョンが必要になった背景・課題}

### 1.2 目的
{このバージョンで達成したいこと}

### 1.3 成功基準
{どうなれば完了と言えるか}

## 2. スコープ

### 2.1 スコープ内
{このバージョンで対応する範囲}

### 2.2 スコープ外
{このバージョンでは対応しない範囲（明示的に除外）}

## 3. 機能一覧

| # | 機能名 | 概要 | 優先度 | ステータス | 詳細設計 |
|---|--------|------|--------|-----------|----------|
| 1 | {機能名} | {簡潔な説明} | MUST/WANT | 未着手/設計中/実装中/完了 | [リンク](../../functions/{大機能}/{中機能}.md) or `-` |

### 優先度の定義
- **MUST**: このバージョンで必ず実装する
- **WANT**: 余裕があれば実装する
- **TBD**: 検討中（スコープ確定前）

### ステータスの定義
- **未着手**: 詳細設計未実施
- **設計中**: 詳細設計作成中（feature-spec実行中）
- **実装中**: 実装作業中
- **完了**: 実装・テスト完了

## 4. 全体制約条件

### 4.1 技術的制約
{技術スタック、既存システムとの整合性など}

### 4.2 ビジネス制約
{期限、リソース、依存関係など}

## 5. 未確定事項

{決定が必要だが、まだ確定していない事項}
- [ ] {未確定事項1}
- [ ] {未確定事項2}
```

## 実行手順

1. **入力情報の確認**: ユーザーから提供された情報を確認
2. **既存ドキュメント参照**:
   - 前バージョンの要件定義があれば参照
   - `docs/versions/` 配下の構造を確認
3. **質疑応答**: 不明点があればAskUserQuestionで確認
   - 特に「スコープ外」の明確化が重要
4. **ドキュメント生成**: テンプレートに沿ってMarkdownファイルを作成
5. **SUMMARY.md更新**: `docs/SUMMARY.md` を更新

## feature-specとの連携

このSkillで作成した機能一覧が、詳細設計（feature-spec）の入力となる。

```
/requirements 実行
  └─ 機能一覧に「記事作成」を定義
       ↓
/feature-spec 実行
  └─ 「記事作成」の詳細仕様を docs/functions/articles/creation.md に作成
       ↓
機能一覧のステータスを「設計中」→「実装中」に更新
```

## 注意事項

- 機能一覧の「詳細設計」列は、feature-spec実行後にリンクを追記する
- 未着手の機能は詳細設計リンクを `-` で表記
- スコープ外は明示的に記載し、後の認識齟齬を防ぐ
- 未確定事項は放置せず、決定期限を意識する
