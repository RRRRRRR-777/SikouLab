# Pre-Commit Review Skill

コミット前の品質チェックを Phase 順に自律実行し、レビューを提示する。

## Step 0: 変更スコープ検出

`git diff --name-only HEAD` で変更ファイルを取得し、以下のフラグを設定する。

| フラグ | 条件 |
|--------|------|
| `BACKEND` | `backend/` 配下に変更あり |
| `FRONTEND` | `frontend/` 配下に変更あり |
| `MIGRATION` | `backend/migrations/` 配下に変更あり |

該当しないフラグの Phase はスキップする。

引数にチケットIDがある場合はレビュー提示時のタイトルに使用する。

## Phase 1: CLI自動チェック

**失敗時は即停止し、修正後に Phase 1 を最初から再実行する。最大リトライ: 3回。**

### Backend（`BACKEND` フラグ時）

以下を順番に実行する。

1. `make -C backend lint`
2. `make -C backend test`（カバレッジ80%以上を確認）
3. `make -C backend build`

### Frontend（`FRONTEND` フラグ時）

以下を順番に実行する。

1. `make -C frontend lint`
2. `make -C frontend test`（カバレッジ80%以上を確認）
3. `make -C frontend build`

### Migration（`MIGRATION` フラグ時）

1. `make -C backend migrate-up`

### ログ確認

- `BACKEND` フラグ時: `/tmp/backend.log` の末尾を確認し、エラーがないことを検証
- `FRONTEND` フラグ時: `/tmp/frontend.log` の末尾を確認し、エラーがないことを検証

### カバレッジ記録

テスト変更がある場合、Phase 1 実行前と実行後のカバレッジを記録する（Phase 4 のレビュー提示で使用）。

## Phase 2: Skill実行

**指摘があり修正した場合は Phase 1 を再実行する。**
**スキル一覧に名前が見当たらなくても、必ず Skill ツールで実行を試みること。一覧は動的に変わるため、実行エラー時のみ「存在しない」と判断する。**

1. `/simplify` を実行 — コードの再利用性・品質・効率をレビュー
2. `/security-review` を実行 — セキュリティ脆弱性を検出

## Phase 3: Playwrightブラウザチェック（`FRONTEND` フラグ時のみ）

**前提**: フロントエンド・バックエンドが起動済みであること。未起動の場合はユーザーに起動を依頼する。

### 3-1: 認証

1. Playwright MCP でログイン画面を開く
2. ユーザーにOAuth操作を依頼する（Claudeは操作しない）
3. 認証完了後、ダッシュボードに遷移することを確認

### 3-2: 変更画面の確認

変更対象の画面について以下を実施する。

#### デスクトップ確認（1280x800）
1. 変更した画面に遷移
2. スクリーンショットを取得
3. 表示崩れ・機能動作を確認

#### モバイル確認（375x812）
1. ブラウザをリサイズ
2. スクリーンショットを取得
3. レスポンシブ表示を確認

#### ダークモード確認
1. テーマをダークモードに切り替え（設定画面またはシステム設定）
2. スクリーンショットを取得
3. コントラスト・視認性を確認

### 3-3: フォーム操作（該当する場合）

1. フォームに入力・送信
2. toast通知の表示を確認
3. データの永続化を確認

### 3-4: エラーチェック

各操作後にブラウザコンソールとログファイルを確認する。エラー発見時は即修正し、Phase 1 から再実行する。

## Phase 4: レビュー提示

以下のフォーマットでレビューを提示する。

```
## [チケットID] チケット名

### 概要
（このチケットで何を・なぜ変更したかを2〜3行で説明）

### 変更ファイル一覧
- path/to/file — 変更の目的

### カバレッジ

（テスト変更時のみ。Phase 1 で記録した変更前後のカバレッジを表形式で提示）

┌────────────┬────────┬────────┐
│ パッケージ　  │  変更前 │  変更後 │
├────────────┼────────┼────────┤
│ xxx        │ xx.x%  │ xx.x%  │
└────────────┴────────┴────────┘

### 各ファイルの変更内容

**path/to/file**
- 変更点1
- 変更点2

### 動作確認（Claude実施済み）

- [x] 単体テスト全件パス
- [x] カバレッジ基準達成（80%以上）
- [x] ビルド成功
- [x] リンター通過
- [x] APIレスポンス確認（該当する場合）
- [x] バックエンドログにエラーなし
- [x] フロントエンドログにエラーなし
- [x] マイグレーション適用確認（該当する場合）
- [x] `/simplify` 実行済み
- [x] `/security-review` 実行済み
- [x] Playwrightブラウザチェック完了（FRONTEND時のみ）

### スクリーンショット（FRONTEND時のみ）

Phase 3 で取得したスクリーンショットのパスを記載する。

### 動作確認（ユーザー実施）

- [ ] OAuth/外部サービス連携の動作確認
- [ ] デザイン（Pencil）との整合性確認
- [ ] UXフローの通し確認
```

## Phase 5: ユーザー確認 → コミット

1. ユーザーの承認を待つ
2. 承認後、`/commit` skillでコミットする
3. チケットIDが指定されている場合、Jiraステータスを「レビュー中」に変更する

## エラーフロー

| 状況 | アクション |
|------|----------|
| Phase 1 失敗 | 原因を修正 → Phase 1 を再実行 |
| Phase 2 で指摘あり | 修正 → Phase 1 を再実行 |
| Phase 3 でエラー発見 | 修正 → Phase 1 から再実行 |
| 各Phase 最大リトライ | 3回。超過時はユーザーに状況を報告し判断を仰ぐ |

## 修正時のチケットコメントルール

チケットIDが指定されている場合、各Phaseで修正が発生した際にJiraチケットへコメントする。

- **修正前**: 修正方針をコメント（何を・なぜ・どう直すか）
- **修正後**: 修正内容と効果・結果をコメント（何を変えたか・テスト結果等）
