# ドキュメント戦略

## 1. 目的
本ドキュメントは、SikouLabプロジェクトにおけるドキュメント作成・管理・運用の方針を定義する。
開発者およびAIエージェントが迷わず、統一された方法でドキュメントを扱えるようにすることを目的とする。

## 2. ドキュメント構造

### 2.2 各ドキュメントの役割

#### SUMMARY.md（インデックス）
- **目的**: docs配下の全ドキュメントの概要を一覧で把握
- **対象読者**: 開発者
- **更新タイミング**: docs配下のマークダウンファイル追加・更新時
- **配置場所**: docs直下

#### development_guidelines.md（開発ガイドライン）
- **目的**: 技術スタック、CI/CD、テスト方針などの開発ルールを定義
- **対象読者**: 開発者
- **更新タイミング**: 開発プロセス変更時
- **配置場所**: docs直下

#### documentation_guidelines.md（本ドキュメント）
- **目的**: ドキュメント作成・管理・運用の方針を定義
- **対象読者**: 開発者
- **更新タイミング**: ドキュメント戦略変更時
- **配置場所**: docs直下

#### versions/（バージョン別ドキュメント）
- **目的**: バージョンごとの要件定義、設計書を管理
- **命名規則**: `a_b_c`形式（例: 1_0_0）
- **対象読者**: 全ステークホルダー
- **更新タイミング**: バージョンアップ時に新ディレクトリ作成
- **含まれるドキュメント**:
  - requirements_definition_document.md（要件定義書）
  - system_design.md（基本設計書）
  - er_diagram.md（ER図）

#### functions/（機能別詳細仕様）
- **目的**: 機能ごとの詳細仕様を管理
- **粒度**: 中機能単位（例: 記事作成、記事編集、記事公開など）
- **対象読者**: 開発者
- **更新タイミング**: 機能実装時・仕様変更時
- **ディレクトリ構成**: 大機能（articles, news等）配下に中機能ごとのファイルを配置
- **例**:
  - articles/creation.md（記事作成機能の詳細仕様）
  - news/translation.md（ニュース翻訳機能の詳細仕様）

#### daily/（議事録）
- **目的**: 日々の議論や意思決定の記録
- **命名規則**: `MMDD.md`（例: 0107.md）
- **対象読者**: プロジェクトメンバー
- **更新タイミング**: ミーティング後

#### adr/（Architecture Decision Records）
- **目的**: 重要なアーキテクチャ上の意思決定を記録
- **命名規則**: `NNNN-title.md`（例: 0001-use-clean-architecture.md）
- **対象読者**: 開発者、アーキテクト
- **更新タイミング**: アーキテクチャ上の重要な決定時

## 3. ファイル命名規則

### 3.1 基本ルール
- **形式**: `snake_case.md`（小文字、アンダースコア区切り）
- **言語**: 英語を基本とする（日本語の場合はローマ字表記も検討）
- **拡張子**: `.md`（マークダウン形式）

### 3.2 特殊なケース
- **議事録**: `MMDD.md`（月日の数字4桁）
- **ADR**: `NNNN-title.md`（連番4桁 + ハイフン + タイトル）
- **バージョンディレクトリ**: `X_Y_Z`（アンダースコア区切り、例: 1_0_0）
- **機能ディレクトリ**: 大機能名（複数形、小文字）配下に中機能ファイル（snake_case.md）

## 4. ドキュメント更新ルール

### 4.1 SUMMARY.md更新ルール（必須）
docs配下のマークダウンファイル（*.md）が追加・更新された際は、**必ず**SUMMARY.mdを更新する。

**記載形式:**
```markdown
* ファイルパス
  * 概要: 2〜3行でそのファイルの内容が簡潔に正確にわかる説明
```

### 4.2 情報探索ルール
1. まずSUMMARY.mdの概要を確認
2. 概要から情報に辿り着きそうであれば、該当するファイルを参照
3. AIエージェントは常にこのフローに従う

### 4.3 ドキュメント作成時の注意
- **目的を明確に**: 何のためのドキュメントかを冒頭に記載
- **対象読者を意識**: 誰が読むかを想定して記述
- **最新性を保つ**: 実装と乖離しないよう定期的に更新
- **TBD（To Be Determined）を活用**: 未決定事項は明示的にTBDと記載

## 5. ドキュメント種別とテンプレート

### 5.1 バージョン管理されるドキュメント（versions/X_Y_Z/）
- requirements_definition_document.md（要件定義書）
- system_design.md（基本設計書）
- er_diagram.md（ER図）
- 詳細設計書（必要に応じて）

### 5.2 バージョン管理されないドキュメント（docs直下）
- development_guidelines.md（開発ガイドライン）
- documentation_guidelines.md（本ドキュメント）

### 5.3 機能別詳細仕様（functions/）
- 中機能単位でファイルを作成
- 大機能ディレクトリ配下に配置
- 例: articles/creation.md, news/fetching.md, stocks/chart.md等

### 5.4 補助ドキュメント
- daily/（議事録）
- adr/（Architecture Decision Records）
- API仕様書（OpenAPI）※配置場所はTBD
- トラブルシューティング（必要に応じて）
- オンボーディングガイド（必要に応じて）
