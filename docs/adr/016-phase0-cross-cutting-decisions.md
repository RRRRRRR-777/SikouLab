# ADR-016: Phase 0 横断的決定事項

## ステータス
承認済み

## コンテキスト
v1.0.0の実装フェーズ（Phase 0〜10）に入るにあたり、全機能に共通する横断的なTBD項目を一括決定する必要がある。
これらはPhase 0（共通基盤構築）の作業開始前に確定させる前提条件であり、後続の全Phaseに影響する。

対象のTBD項目:
- D-01: ログ出力方針（ライブラリ選定）
- D-02: ログレベル方針
- D-03: スケルトン表示
- D-04: エラー表示
- D-05: Pencil未定義画面の対応

## 決定

### D-01: ログ出力方針 → zerolog

**zerolog（構造化JSON Logger）** を採用する。

```bash
go get -u github.com/rs/zerolog
```

### D-02: ログレベル方針 → 標準4レベル

| レベル | 用途 | 例 |
|--------|------|-----|
| **ERROR** | 即時対応が必要な異常 | DB接続失敗、外部API障害、パニックリカバリ |
| **WARN** | 注意が必要だが動作は継続 | レート制限接近、非推奨API使用、リトライ発生 |
| **INFO** | 主要イベントの記録 | リクエスト処理、ユーザー認証、バッチ実行 |
| **DEBUG** | 開発・デバッグ時のみ | SQL実行詳細、変数値、処理ステップ |

- 本番環境: INFO以上を出力
- 開発環境: DEBUG以上を出力

### D-03: スケルトン表示 → あり

読み込み中はshadcn/uiの`Skeleton`コンポーネントでスケルトンUIを表示する。

- 全画面共通の方針として適用
- 体感速度の向上を目的とする

### D-04: エラー表示 → トースト

エラーメッセージはトースト（画面隅の一時表示）で通知する。

- shadcn/uiの`Toast`（Sonner）コンポーネントを使用
- 操作フローを邪魔しないUIを優先
- フォームバリデーションエラーはインラインで表示（React Hook Form + Zodの標準動作）

### D-05: Pencil未定義画面の対応 → テキストベースで進める

Pencilデザインが未定義の画面（S-04, S-05, S-07, S-08, S-09, S-11, S-12, S-15, S-16, S-18〜S-25）は、詳細設計書の仕様をベースに実装し、後からデザイン調整を行う。

- 開発速度を優先
- 既存のPencil定義済み画面（S-01, S-02, S-03, S-06, S-10, S-14, S-17）のデザインパターンを踏襲

## 検討した選択肢

### D-01: ログライブラリ比較

| ライブラリ | ns/op | allocs/op | 外部依存 | JSON出力 | 学習コスト | 将来性 |
|-----------|------:|----------:|:--------:|:--------:|:----------:|:------:|
| **zerolog** | 30 | 0 | あり（依存ゼロ） | ◎ ネイティブ | ◎ 低い | ○ |
| slog | 174 | 0 | なし（Go標準） | ○ Handler経由 | ◎ 低い | ◎ |
| zap | 71 | 0 | あり | ○ Handler経由 | △ デュアルAPI | ○ |

### D-04: エラー表示比較

| 方式 | UX | 実装コスト | 適用場面 |
|------|:--:|:----------:|----------|
| **トースト** | ◎ | ○ | API系エラー全般 |
| インライン | ○ | ○ | フォームバリデーション |
| 両方 | ◎ | △ | 場面に応じて使い分け |

## 理由

### D-01: zerologを採用した理由
1. **パフォーマンス**: 30 ns/opで最速。SicouLabの規模では実用上の差はないが、コストなく得られる性能
2. **JSON特化**: Cloud Loggingとの親和性が高く、構造化ログをネイティブに出力（ADR-007の方針と整合）
3. **学習コスト**: チェーンAPIが直感的で習得が容易
4. **依存の軽さ**: zerolog自体の外部依存がゼロ。`go.sum`に追加パッケージが入らない
5. **slogを不採用とした理由**: パフォーマンス差は実用上無視できるが、zerologのJSON特化・チェーンAPIが開発体験で優位。外部依存のリスクは極めて低い

### D-02: 標準4レベルを採用した理由
- 業界標準の運用パターンに準拠
- Cloud Logging/Cloud Monitoringのseverityマッピングと一致
- ERRORとWARNの明確な区別により、アラート設計が容易

### D-03: スケルトン表示を採用した理由
- 体感速度を向上させるUXベストプラクティス
- shadcn/uiに組み込みコンポーネントがあり追加コスト最小

### D-04: トーストを採用した理由
- 操作フローを中断しないUI
- shadcn/uiのSonner統合で実装が容易
- フォームバリデーションはReact Hook Form + Zodの標準インラインエラーで対応

### D-05: テキストベースを採用した理由
- 開発速度を優先（Pencilデザイン作成の待ち時間を排除）
- 定義済み画面のパターンを踏襲することで一貫性を維持

## 影響
- **バックエンド全体**: zerolog + 4レベルログをミドルウェア・ハンドラ・ユースケース層で統一適用
- **フロントエンド全体**: Skeleton・Toast・Sonnerコンポーネントを共通基盤として導入
- **ADR-007（ログ監視）**: zerologのJSON出力がCloud Loggingに直接対応。追加のフォーマット変換不要
- **Phase 0作業**: これらの決定に基づいて共通基盤を構築開始可能
