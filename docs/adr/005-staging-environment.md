# ADR-005: 環境戦略

## ステータス
承認済み

## コンテキスト

### staging環境の要否
本番デプロイ前の確認環境としてstaging環境の必要性を検討。

現状の運用想定：
- QA担当による本番リリース前の最終確認: なし
- 外部サービス（Stripe等）との結合テスト: モック化で対応
- 本番と同等のデータ量でのテスト: すぐには必要なし

### dev環境の設計
[ADR-015](./015-deploy-platform.md)でCloud Runの採用が決定し、インフラ設計書で本番環境の構成が定まった。開発・検証用のdev環境をどのような方針で設計するかの決定が必要。

考慮事項：
- 一人運用のため、dev環境の常時稼働は不要
- prod環境との構成差異が大きいと、環境固有の問題を見逃すリスクがある
- dev環境のコストは最小限に抑えたい

## 決定

### staging環境
**初期はstaging環境を構築せず、必要に応じて後から追加する。**

### dev環境
**prod環境と同等構成のdev環境をオンデマンドで起動/停止する。ライフサイクルはCD（GitHub Actions）で管理する。**

環境構成: dev → prod 直接（stagingなし）

### dev環境の設計方針

#### オンデマンドライフサイクル

dev環境のGCPリソースはGitHub Actionsワークフローで作成・削除する。停止時は全リソースを削除し、コストをゼロにする。

| 操作 | トリガー | 内容 |
|------|---------|------|
| 起動（`dev-up`） | 手動実行 or devブランチpush | GCPリソース作成 → デプロイ |
| 停止（`dev-down`） | 手動実行 | GCPリソース全削除 |

#### prod環境との差分

dev環境はprodと同等の構成を維持しつつ、以下の項目のみ縮小する。

| 項目 | prod | dev | 差分理由 |
|------|------|-----|---------|
| Cloud Run min-instances | FE=1, BE=0 | **全て0** | 常時待機不要 |
| Cloud Run max-instances | 3 | **1** | 負荷テスト不要 |
| Cloud SQL PITR | 有効 | **無効** | テストデータに復旧保証不要 |
| Cloud SQL バックアップ保持 | 7日 | **3日** | コスト削減 |
| Cloud Armor | フルポリシー | **最小限**（レート制限のみ） | WAF/地域制限はprod検証で十分 |
| 監視アラート | フル（10件） | **最小限**（サービスダウン・バッチ失敗のみ） | 開発中は手動確認で十分 |
| Uptime Check | あり | **なし** | 常時監視不要 |
| 予算アラート | $91基準 | **$40基準** | dev環境のコスト上限管理 |

#### dev環境で同等に維持する項目

以下はprodとの構成差異による問題を防ぐため、同等構成を維持する。

- Cloud Load Balancing（URL Map + Serverless NEG）
- SSL/TLS（Google Managed SSL）
- カスタムドメイン（`dev.sicoulab.com` 等）
- Secret Manager（環境別シークレット）
- Cloud Logging（構造化ログ、7日保持）
- Artifact Registry（共有、タグで環境区別）

#### 環境分離

| リソース | 分離方式 |
|---------|---------|
| GCPプロジェクト | **同一プロジェクト**（リソース名のプレフィックスで区別） |
| Cloud Run サービス名 | `sicoulab-frontend-dev` / `sicoulab-api-dev` |
| Cloud SQL インスタンス名 | `sicoulab-db-dev` |
| Secret Manager | `dev-database-url` 等、環境プレフィックス付き |
| Cloud Storage バケット | `sicoulab-assets-dev` |
| Cloud Armor ポリシー | `sicoulab-armor-dev` |
| ドメイン | `dev.sicoulab.com`（サブドメイン） |

#### dev環境のコスト試算

| 状態 | 月額（USD） | 内訳 |
|------|-----------|------|
| **停止時** | **$0** | 全リソース削除済み |
| **稼働時** | **$30-50** | Cloud Run（min=0）+ LB + Armor（最小）+ Cloud SQL + Storage |

## 検討した選択肢

### staging環境

| 評価軸 | dev → prod直接 | staging経由 |
|--------|---------------|-------------|
| 運用コスト | ◎ 環境1つ分削減 | △ 追加のインフラ費用 |
| デプロイ速度 | ◎ 速い | ○ 1ステップ増 |
| リスク管理 | △ 本番で初めて発覚する問題あり得る | ◎ 本番前に検出 |
| 初期構築コスト | ◎ 不要 | △ 環境構築が必要 |
| 後から追加 | ◎ 必要になったら追加可能 | - |

### dev環境のライフサイクル

| 方式 | コスト | prod同等性 | 運用負荷 |
|------|--------|-----------|---------|
| 常時稼働 | × $30-50/月が常時 | ◎ 常にprodと同じ | ◎ 意識不要 |
| **オンデマンド（CD管理）** | **◎ 停止時$0** | **◎ 起動時はprod同等** | **○ 起動/停止のワークフロー実行** |
| ローカルのみ | ◎ $0 | × 環境差異大 | ◎ 意識不要 |

## 理由

### staging不要の理由
1. **現状の運用に不要**: QA担当なし、外部サービスはモック化
2. **コスト最小化**: 初期段階では環境構築・運用コストを抑える
3. **後から追加可能**: 必要性が生じた時点で追加できる

### dev環境オンデマンド方式の理由
1. **コスト最適**: 停止時$0。開発集中期間のみ稼働させることで、月額コストを最小限に抑える
2. **prod同等性**: LB・Armor・SSL含むprodと同等構成により、環境固有の問題を早期検出できる
3. **CD管理の利点**: 環境構築手順がコード化され、再現性が高い。手動構築の属人化を防ぐ
4. **一人運用に最適**: 開発しない時間帯（大半の時間）はコストゼロ。必要な時にワンクリックで起動

### Cloud Armor最小限の理由
- dev環境は外部公開されるが、本番トラフィックは流れない
- レート制限のみで不正アクセスの基本防御は確保
- WAFルール（OWASP CRS）や地域制限はprod環境で検証すれば十分
- ポリシー数削減でコストを$5→$5（ポリシー自体は最低$5だが、ルール数を最小化）

## 見直しトリガー

### staging追加を検討するタイミング
- QA担当が参加し、本番リリース前確認が必要になった時
- 本番と同等のデータ量でのテストが必要になった時
- 複数チームの変更を統合してテストする必要が生じた時

### dev環境の常時稼働へ移行するタイミング
- 開発頻度が上がり、起動/停止の運用コスト（待ち時間・手間）が無視できなくなった時
- 複数人が同時にdev環境を利用する必要が生じた時
- CI/CDパイプラインでdevデプロイが日常的に必要になった時

> 常時稼働に移行する際は、リソーススペック（min-instances、max-instances、Cloud SQL マシンタイプ等）をその時点の利用状況に基づいて再検討する。

## 影響
- CI/CDパイプラインは dev → prod の2環境で構築
- staging追加時はdev環境のCDワークフローを複製して対応可能
- GitHub Actionsに `dev-up.yml` / `dev-down.yml` ワークフローを追加
- Secret Managerに環境別シークレットを登録
- DNS設定に `dev.sicoulab.com` サブドメインを追加
- インフラ設計書に環境別差分セクションを追加（[infrastructure.md](../versions/1_0_0/infrastructure.md)）
- ADR-006（デプロイフロー）: devブランチへのデプロイフローを追記
